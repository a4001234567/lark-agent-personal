# 个人助理向MCP服务器

我们面向单人使用场景，设计了基于MCP协议的飞书agent接入框架，它安全**支持第三方插件可控使用飞书接口**。

## 简单描述

我们考虑服务单用户的agent场景，其中既包括个人使用，也包含企业中单个员工使用的场景。
这一场景下，有特殊的工具需求与侧重范围（主要集中在日程、任务、文档等功能），并且需要与企业中其他环境有一定隔离性。
我们因此以校园场景下单个学生用户为设计考量。

我们在功能上进行了重新组合包装以**优化工具逻辑**，针对单人场景进行功能重新设计与skill重写，同时调整了工具处理逻辑（例如**修复了飞书官方工具在重复日程中的处理失败**）以更好响应个人需求（例如课程、周期性会议的临时调整）。同时更广泛地支持事件回调以及飞书卡片。

我们搭建的MCP服务器与cli框架相比，可以**持久化维持用户授权状态**，并且**支持其他第三方工具安全可控便捷地调用飞书能力**（例如本仓库中的课表导入、生活轨迹同步等功能使用飞书日程授权），与agent共享授权将简化流程。

> [!Important]
> 我们**首次发现并验证**了飞书 Schema 2.0 **互动卡片提示词隐写**方法，攻击者**无需任何权限**，只需构造一张外观正常的卡片，当用户转发至 LLM Agent 时--无任何感知，即可实现隐藏的提示词注入。
> 
> 这一攻击方法对飞书官方实现和我们的MCP服务器均可生效，尽管部分思路存在局限性，但根据社会工程学精心构造的卡片仍然能够较好的攻击目标
>

## MCP服务器
构建面向单用户的**个人助理**，不管是企业场景还是个人场景，使用有限的能力（目前主要包括消息、日程、任务、文档），可以将任何AI harness接入飞书。
在有限的工具范围内，我们根据实际场景优化了工具的组合形式、提示词、以及编排：

> 以日程为例，官方插件包括9个事件管理、3个日历管理、2个参会人管理、2个忙闲查询插件，基本为飞书API的直接重新包装。
> 我们将事件精简为4个插件，涵盖创建、检索/列出、修改、删除的基本原子操作，并且在skill中给出局限性。
> 针对飞书API不能直接修改重复日程中单个实例的问题，我们对API进行了重新组合，通过执行多个API call组合成真正鲁棒可用的工具，结合skill能够满足常规需求。
> 如需适配多人场景也可以此为基础构建精简版本工具。
> 同时优化输出格式以精简token消耗。

## 可扩展性
飞书与其他channel不同之处在于其丰富的工具支持，这也带来其他第三方数据与操作的适配问题，我们希望用户可以通过MCP服务器实现接入其他的第三方数据源。
在这一项目中，我们以课表导入日程、佳明健康数据同步、基于日程安排的身心健康分析为例。通过安装skill，并调用MCP服务器的auth token，可实现无感知的访问，避免重复授权。

## 隐写注入原理与潜在危害

飞书 Schema 2.0 互动卡片支持丰富的结构化数据，包含多语言、图片、按钮等多种元素。这些元素中存在特定字段：
- 对于人类用户：在正常使用中永远不会显示
- 对于飞书服务器：在通过 WebSocket 下发事件或 API 返回卡片内容时，会完整保留这些字段

当 LLM Agent 通过 API 或 WebSocket 读取卡片内容时，如果直接将整个卡片结构序列化后送入 LLM 上下文，这些隐藏字段就会和正常内容一起被 LLM 读取，攻击者可以利用这一点注入任意指令。

我们发现了潜在的上述特性，并提出了以下的攻击链路：

- 攻击者构造外观正常的卡片，诱导用户转发给 AI Agent
- 隐藏指令被 Agent 理解为用户指令，执行未经授权操作：
  - 批量导出敏感数据并发送至第三方
  - 访问其他特定恶意内容
  - 删除或修改数据
  - 访问恶意链接获取认证信息

并进行了实际验证。
